How should dbt + Spark integrate?

This is the core architectural decision. Each approach has a different learning focus.

A

dbt-spark adapter → Spark Thrift Server → PostgreSQL

dbt uses the dbt-spark adapter and talks to a Spark Thrift Server (JDBC). Spark reads raw files, runs all dbt models, stores results in Delta/Parquet. A final export step pushes mart tables to PostgreSQL for Superset.

Pros

  • Most "authentic" dbt + Spark experience
  • All dbt models run on Spark SQL
  • Learn dbt-spark adapter patterns

Cons

  • Thrift Server adds Docker complexity
  • Slightly harder to debug
B

Spark for ingestion, dbt-postgres for transforms

A PySpark ingestion script reads CSVs + Postgres source schemas and loads raw data into a PostgreSQL raw schema. Then dbt uses the postgres adapter to run all Medallion / Data Vault / SCD logic in Postgres.

Pros

  • Simplest Docker setup
  • dbt-postgres adapter is most mature
  • Easier to debug transforms

Cons

  • dbt doesn't actually touch Spark — less Spark/dbt integration practice
C

Spark Connect (Spark 3.4+) → dbt-spark → PostgreSQL

Use the modern Spark Connect protocol instead of Thrift Server. A lightweight Spark Connect server runs in Docker; dbt-spark connects to it. Final mart tables are written to PostgreSQL for Superset. Cleaner than Thrift but newer.

Pros

  • Modern Spark architecture (Spark 3.4+)
  • Lighter than Thrift Server
  • Full dbt + Spark integration

Cons

  • dbt-spark + Spark Connect is still maturing
  • Fewer community examples for learning

My recommendation

Option A — dbt-spark via Thrift Server is the most widely documented pattern for learning. More tutorials, more examples, and you get genuine dbt+Spark integration experience. The Thrift Server Docker image is well-established.
