How should Spark and PostgreSQL work together?

This shapes the entire architecture. Pick the role that matches what you want to learn.

A

Spark processes, PostgreSQL stores

Raw data → Spark (dbt runs Spark SQL) → transformed results written to PostgreSQL → dashboard reads Postgres. Spark is the workhorse; Postgres is the final serving layer.

Pros

  • Realistic data engineering pipeline
  • Learn both Spark and Postgres
  • Dashboard queries are fast (Postgres)

Cons

  • More moving parts to configure
B

Spark only (with Spark's built-in warehouse)

Raw data → Spark processes everything → dbt models stored in Delta Lake / Spark warehouse → dashboard reads Spark directly. PostgreSQL not used.

Pros

  • Simpler — one engine, one storage
  • Best for learning dbt + Spark together

Cons

  • No Postgres experience
  • Dashboard setup is more complex
C

PostgreSQL primary, Spark for heavy transforms

dbt runs mostly against PostgreSQL; Spark is called only for specific heavy transformations. Most of the Data Vault / SCD Type 2 logic lives in Postgres.

Pros

  • Simpler dbt setup (postgres adapter is mature)
  • Easier to debug and iterate

Cons

  • Less Spark practice
  • Spark feels bolted-on
