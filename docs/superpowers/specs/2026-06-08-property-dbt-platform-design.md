# Property Consumer Data Platform ‚Äî Design Spec

**Date:** 2026-06-08  
**Status:** Approved for implementation

---

## Context

This is a learning project for dbt Core, covering:
- **dbt-core** with the dbt-postgres adapter
- **Apache Spark** (PySpark) as the ingestion / ETL engine
- **Data Vault 2.0** (light approach: Hubs, Links, Satellites with hashdiff-based SCD Type 2)
- **Medallion Architecture** (Bronze ‚Üí Silver ‚Üí Gold ‚Üí Mart)
- **Apache Superset** for dashboarding
- Everything runs locally via **Docker Compose**

The business domain is **consumer property** ‚Äî property transactions and attributes, agent/owner data.

## Data Sources

### Source 1 ‚Äî CSV (`property_raw.csv`)
Generated by a **Python/Faker script** (`scripts/generate_data.py`).  
Single file containing **both** transaction events and property attributes (mixed columns).  
Split into separate entities in the Silver/staging layer by dbt.

| Field group | Fields |
|---|---|
| Transaction | `transaction_id`, `property_id`, `sale_price`, `sale_date`, `buyer_id`, `seller_id`, `agent_id`, `sale_type`, `settlement_date`, `days_on_market` |
| Property attributes | `property_id`, `address`, `suburb`, `postcode`, `state`, `bedrooms`, `bathrooms`, `land_size_sqm`, `property_type`, `year_built`, `listing_price` |

Scale: ~10,000 transaction rows ¬∑ ~2,000 distinct properties

### Source 2 ‚Äî PostgreSQL schema `src_crm`
Seeded by the same Faker script directly into the `src_crm` schema.

**`src_crm.owners`**: `owner_id`, `full_name`, `email`, `phone`, `suburb`, `state`, `registration_date`, `investor_flag`  
**`src_crm.agents`**: `agent_id`, `full_name`, `email`, `agency_name`, `license_no`, `region`, `accreditation_tier`, `active_from`

Scale: ~500 owners ¬∑ ~100 agents

## Medallion Architecture

### Bronze ‚Äî `bronze` schema
- Loaded by PySpark, not dbt
- Exact 1:1 copy of sources, no transformations
- Append-only; includes `_loaded_at` timestamp column added by Spark
- Tables: `bronze.property_raw`, `bronze.src_crm_owners`, `bronze.src_crm_agents`

### Silver ‚Äî `silver` schema (dbt staging models)
- dbt models in `models/staging/`
- **Key transform**: splits `bronze.property_raw` into `stg_transactions` and `stg_properties`
- Casts all types, standardises nulls, adds `record_source` column
- Computes `hash_key` (MD5 of business key) and `hashdiff` (MD5 of all attribute columns) for each entity
- Models: `stg_transactions`, `stg_properties`, `stg_owners`, `stg_agents`

### Gold ‚Äî `gold` schema (Data Vault 2.0 light)
- dbt models in `models/vault/`
- **Hubs** (business keys + metadata only):
  - `hub_property` ‚Äî `property_hk`, `property_id`, `load_date`, `record_source`
  - `hub_owner` ‚Äî `owner_hk`, `owner_id`, `load_date`, `record_source`
  - `hub_agent` ‚Äî `agent_hk`, `agent_id`, `load_date`, `record_source`
  - `hub_transaction` ‚Äî `transaction_hk`, `transaction_id`, `load_date`, `record_source`
- **Links** (relationships between hubs):
  - `lnk_transaction_property` ‚Äî `transaction_hk`, `property_hk`
  - `lnk_transaction_buyer` ‚Äî `transaction_hk`, `owner_hk` (buyer)
  - `lnk_transaction_seller` ‚Äî `transaction_hk`, `owner_hk` (seller)
  - `lnk_transaction_agent` ‚Äî `transaction_hk`, `agent_hk`
- **Satellites** (attributes + SCD Type 2 via `load_date` / `load_end_date`):
  - `sat_property_details` ‚Äî property attributes; `hashdiff` detects changes ‚Üí SCD2 rows
  - `sat_owner_details` ‚Äî owner attributes; SCD2
  - `sat_agent_details` ‚Äî agent attributes; SCD2
  - `sat_transaction_details` ‚Äî transaction financials; append-only (transactions don't change)

### Mart ‚Äî `mart` schema (dbt mart models)
- dbt models in `models/mart/`
- Joins vault tables into flat, Superset-ready tables
- `mart_property_sales` ‚Äî one row per transaction with property + agent + price details
- `mart_agent_performance` ‚Äî aggregated agent metrics (volume, avg price, days on market)
- `mart_suburb_trends` ‚Äî median price + transaction count by suburb + month
- `mart_owner_portfolio` ‚Äî owner-level aggregates (properties owned, investor flag, buy/sell counts)

## Data Vault 2.0 ‚Äî Light Approach

**What's included:**
- Hash keys (MD5) on all business keys
- Hashdiff on all satellite payloads (drives SCD Type 2 detection)
- `load_date` + `load_end_date` on satellites for point-in-time querying
- `record_source` column on all vault tables

**What's excluded (production patterns, not needed for learning):**
- Record Tracking Satellites (RTS)
- Separate content/hashdiff staging tables
- Bridge tables
- Point-in-time (PIT) tables (mart models use simple `load_end_date IS NULL` for current records)

---

## SCD Type 2 Implementation

Implemented in satellite models using dbt's incremental strategy:

1. Incoming staging row has a `hashdiff` computed over all attribute columns
2. If `hashdiff` differs from the latest satellite row for that hub hash key ‚Üí insert new row with new `load_date`, set previous row's `load_end_date`
3. Current record = `load_end_date IS NULL`
4. Mart models use `WHERE load_end_date IS NULL` to get current state

## Orchestration ‚Äî Apache Airflow

Airflow runs a single DAG (`property_pipeline`) that sequences the full pipeline on a daily schedule.

**DAG tasks:**
1. `generate_and_ingest` ‚Äî runs the PySpark job (`spark-ingest` container) via `DockerOperator`; generates Faker data and loads `bronze.*` tables
2. `dbt_run` ‚Äî runs `dbt run` via `BashOperator` inside the `dbt` container
3. `dbt_test` ‚Äî runs `dbt test` via `BashOperator`; task fails and alerts if any test fails

**Key decisions:**
- Airflow uses the same `postgres` service for its metadata DB (separate database `airflow_db`, not a separate container)
- DAG uses `DockerOperator` for the Spark step and `BashOperator` for dbt (simpler than Cosmos for a learning project)
- `astronomer-cosmos` (dbt-native Airflow integration) is excluded ‚Äî it adds complexity without clear learning benefit at this stage

**DAG file location:** `airflow/dags/property_pipeline.py`

## Verification

1. `docker compose up -d` ‚Üí all services start (postgres, airflow, superset)
2. Open Airflow at `http://localhost:8080` ‚Üí trigger `property_pipeline` DAG manually ‚Üí all 3 tasks succeed
3. Check `bronze.*` tables populated in Postgres after `generate_and_ingest` task
4. Check all dbt schemas (`silver`, `gold`, `mart`) populated after `dbt_run` task
5. Verify `dbt_test` task passes with zero failures
6. Manually update a property attribute in `bronze.property_raw`, trigger DAG again ‚Üí verify new SCD2 row in `gold.sat_property_details` with correct `load_date` / `load_end_date`
7. Open Superset at `http://localhost:8088` ‚Üí connect to Postgres mart schema ‚Üí verify all 4 mart tables visible and dashboards load
